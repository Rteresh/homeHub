# HomeHub — архитектура MVP

## 1. Назначение документа

Этот документ фиксирует первый архитектурный шаг разработки HomeHub: структуру проекта, границы компонентов, основные потоки данных и зоны ответственности агентов. Он опирается на `PROJECT_SPEC.md` и профили из `agents/`.

## 2. Подтверждённые решения

- MVP фокусируется на приёме, хранении, просмотре, поиске, скачивании и удалении файлов/медиа.
- Управление умными устройствами, MQTT, WebSocket-интеграции, расширенная аналитика и публичный доступ из интернета относятся к развитию после MVP.
- Основной стек MVP:
  - Python;
  - Django для backend/API, веб-интерфейса, аутентификации и административной панели;
  - Django ORM для метаданных;
  - SQLite для локального MVP;
  - aiogram 3 для Telegram-бота;
  - локальная файловая система для приватного хранения файлов;
  - Docker/Docker Compose для воспроизводимого локального запуска.
- Реальные пути файловой системы не возвращаются клиентам.
- Проверка прав выполняется на backend-уровне для просмотра, превью, скачивания, удаления и изменения метаданных.
- Для публичных URL файлов используется UUID `public_id`, а не числовой database ID.

## 3. Архитектурный стиль

HomeHub строится как Django-монолит с отдельным процессом Telegram-бота. Общая доменная логика живёт в сервисном слое Django-приложений, чтобы веб-интерфейс, API, админка и бот не дублировали правила доступа, сохранения и обработки файлов.

Главный принцип зависимостей:

- UI, API и Telegram handlers вызывают сервисы.
- Сервисы управляют бизнес-правилами, транзакциями и согласованностью файла с метаданными.
- Репозитории/ORM отвечают за чтение и запись данных.
- Storage-адаптер отвечает за безопасные пути, запись, чтение и удаление файлов.
- Низкоуровневые адаптеры не принимают решений о правах доступа.

## 4. Компоненты MVP

### 4.1. Django project

Назначение:
- настройки проекта;
- маршрутизация;
- подключение приложений;
- конфигурация БД, статических файлов, приватного хранилища и логирования.

Предлагаемый пакет:
- `homehub/`

### 4.2. Accounts

Назначение:
- пользователи Django;
- роли `admin` и `user`;
- связь Django-пользователя с Telegram `user_id`/`chat_id`;
- allowlist для доступа через Telegram.

Предлагаемое приложение:
- `apps/accounts/`

Ключевые правила:
- пользователь без привязанного и разрешённого Telegram ID не может загружать файлы через бота;
- роль администратора должна проверяться через Django permissions/groups или явное поле профиля;
- Telegram ID не является единственной моделью авторизации для чувствительных действий.

### 4.3. Files

Назначение:
- модель метаданных файла;
- список файлов;
- фильтрация, сортировка, поиск и пагинация;
- скачивание и удаление с проверкой прав;
- сервисы сохранения и удаления файлов.

Предлагаемое приложение:
- `apps/files/`

Основная модель:
- `FileAsset`

Ключевые поля:
- `id`;
- `public_id`;
- `owner`;
- `source`;
- `original_name`;
- `storage_path`;
- `mime_type`;
- `category`;
- `size_bytes`;
- `checksum`;
- `status`;
- `telegram_file_id`;
- `telegram_message_id`;
- `created_at`;
- `updated_at`.

Индексы для MVP:
- `owner, created_at`;
- `category, created_at`;
- `status, created_at`;
- `checksum` при включении поиска дублей.

### 4.4. Media

Назначение:
- представление фото/видео в виде медиатеки;
- выдача превью через backend с проверкой прав;
- подготовка места для генерации thumbnails.

На старте media может быть частью `apps/files/`, если отдельное приложение создаст лишнюю сложность. Выделение в `apps/media/` оправдано после появления отдельной обработки превью, альбомов или меток.

### 4.5. Bot

Назначение:
- aiogram 3 runtime;
- команды `/files`, `/media`, `/status`, `/help`, `/download <id>`, `/delete <id>`;
- получение файлов из Telegram;
- передача файлов в общий сервис сохранения;
- пользовательские ответы без раскрытия внутренних деталей.

Предлагаемый пакет:
- `bot/`

Внутренние зоны:
- `bot/routers/` — обработчики команд и файлов;
- `bot/middlewares/` — авторизация и контекст пользователя;
- `bot/services/` — тонкие адаптеры к Django-сервисам;
- `bot/runtime/` — запуск long polling для локального MVP.

Для локального MVP выбирается long polling. Webhook можно добавить позже при появлении публичного домена и HTTPS.

### 4.6. Storage

Назначение:
- безопасное формирование внутренних путей;
- запись исходных файлов;
- потоковое чтение для скачивания;
- удаление и cleanup;
- защита от обхода путей и коллизий имён.

Предлагаемый пакет:
- `apps/files/storage.py` или `apps/files/storage/`

Рабочие директории:
- `storage/uploads/`;
- `storage/thumbnails/`;
- `storage/tmp/`;
- `storage/logs/`.

### 4.7. Web UI

Назначение:
- страницы авторизации;
- список файлов;
- фильтры, сортировка, поиск и пагинация;
- медиатека;
- скачивание и удаление через backend endpoints.

Для MVP используются шаблоны Django и адаптивный HTML/CSS. SPA-фреймворк не нужен до появления интерфейсной сложности, которую шаблоны уже не покрывают.

### 4.8. Admin

Назначение:
- управление пользователями;
- просмотр и коррекция метаданных;
- диагностика загрузок;
- базовая статистика.

Django admin используется как быстрый административный интерфейс MVP, но не заменяет пользовательский веб-интерфейс.

## 6. Основные доменные статусы

`FileAsset.status`:
- `pending` — метаданные созданы, файл ещё сохраняется или обрабатывается;
- `ready` — файл сохранён и доступен владельцу/администратору;
- `failed` — загрузка или обработка завершилась ошибкой;
- `deleted` — файл удалён логически, физическое удаление выполнено или запланировано.

`FileAsset.category`:
- `photo`;
- `video`;
- `audio`;
- `document`;
- `archive`;
- `other`.

`FileAsset.source`:
- `telegram`;
- `web`;
- `system`.

## 7. Ключевые сервисы

### 7.1. `FileIngestionService`

Отвечает за загрузку файла из внешнего источника в систему. Принимает владельца, источник, исходное имя, MIME-тип, размер, поток/путь временного файла и внешние метаданные. Создаёт или обновляет `FileAsset`, записывает файл через storage-адаптер и переводит статус в `ready` или `failed`.

Обязательные свойства:
- транзакционная согласованность метаданных;
- cleanup временных файлов при ошибке;
- защита от повторной обработки одного Telegram update там, где доступны внешние идентификаторы;
- понятные ошибки: отказ доступа, неверный тип/размер, ошибка Telegram API, ошибка записи, конфликт дубля.

### 7.2. `FileAccessService`

Отвечает за правила доступа. Принимает пользователя и файл, возвращает разрешение на действие: просмотр, превью, скачивание, удаление или изменение метаданных.

Правило MVP:
- администратор может действовать с любым файлом;
- обычный пользователь может действовать только со своими файлами;
- неизвестный или неактивный пользователь не получает доступ.

### 7.3. `FileQueryService`

Отвечает за списки файлов, фильтры, сортировку, поиск и пагинацию. Не должен возвращать файлы, недоступные текущему пользователю.

### 7.4. `FileStorage`

Отвечает за физическую запись, чтение и удаление файлов. Принимает только нормализованные данные от сервиса, сам не решает права доступа.

### 7.5. `TelegramIdentityService`

Связывает Telegram `user_id`/`chat_id` с Django-пользователем и ролью. Используется middleware/handlers бота перед выполнением команд и загрузок.

## 8. Потоки данных

### 8.1. Загрузка через Telegram

1. Пользователь отправляет файл боту.
2. aiogram router получает update.
3. Middleware определяет пользователя по Telegram ID.
4. Handler валидирует базовый тип update и передаёт управление сервису.
5. Бот скачивает файл во временную директорию или передаёт поток storage-сервису.
6. `FileIngestionService` создаёт метаданные и сохраняет файл.
7. При успехе статус становится `ready`, пользователь получает подтверждение.
8. При ошибке статус становится `failed` или запись откатывается, пользователь получает понятное сообщение.

### 8.2. Список файлов в вебе

1. Пользователь открывает страницу файлов.
2. Django view проверяет аутентификацию.
3. `FileQueryService` применяет owner-scope, фильтры, сортировку и пагинацию.
4. Шаблон получает только доступные текущему пользователю записи.

### 8.3. Скачивание файла

1. Пользователь запрашивает скачивание по `id`.
2. Django view получает `FileAsset`.
3. `FileAccessService` проверяет право скачивания.
4. `FileStorage` открывает внутренний путь.
5. Backend отдаёт файл потоком или через короткоживущий механизм.
6. Реальный путь хранилища не попадает в ответ.

### 8.4. Удаление файла

1. Пользователь или администратор отправляет запрос удаления.
2. `FileAccessService` проверяет право удаления.
3. Сервис переводит запись в `deleted` или удаляет физический файл в согласованной операции.
4. UI или бот получает результат операции.

## 9. API и URL MVP

Минимальные backend endpoints:
- `GET /files/` — страница списка файлов;
- `GET /files/<id>/download/` — скачивание с проверкой прав;
- `POST /files/<id>/delete/` — удаление с проверкой прав;
- `GET /media/` — медиатека;
- `GET /media/<id>/preview/` — превью с проверкой прав.

Если потребуется JSON API:
- `GET /api/files/`;
- `GET /api/files/<id>/`;
- `GET /api/files/<id>/download/`;
- `DELETE /api/files/<id>/`.

JSON API не обязателен для первого локального веб-MVP, если Django views и templates покрывают сценарии.

## 10. Конфигурация

Минимальные переменные окружения:
- `DJANGO_SECRET_KEY`;
- `DJANGO_DEBUG`;
- `DJANGO_ALLOWED_HOSTS`;
- `DATABASE_URL` или отдельные настройки SQLite/PostgreSQL;
- `HOMEHUB_STORAGE_ROOT`;
- `TELEGRAM_BOT_TOKEN`;
- `TELEGRAM_ALLOWED_USER_IDS`;
- `TELEGRAM_ADMIN_USER_IDS`.

Секреты не хранятся в коде и не попадают в логи.

## 11. Логирование и диагностика

Минимальные события:
- старт web/backend;
- старт bot runtime;
- успешная загрузка файла;
- отказ доступа;
- ошибка сохранения файла;
- ошибка Telegram API;
- удаление файла;
- скачивание файла администратором или владельцем.

Логи должны содержать технический контекст без секретов:
- `file_asset_id`;
- `user_id`;
- `telegram_user_id`;
- `telegram_chat_id`;
- `update_id`;
- статус операции.

## 12. Тестовая стратегия MVP

Минимальные автоматические проверки после появления Django-кода:
- модель `FileAsset` создаётся с обязательными полями;
- пользователь видит только свои файлы;
- администратор видит все файлы;
- чужой файл нельзя скачать;
- чужой файл нельзя удалить;
- storage-адаптер не позволяет выйти за пределы `HOMEHUB_STORAGE_ROOT`;
- повторная обработка одного Telegram update не создаёт дубль, если внешний идентификатор доступен.

Ручные проверки:
- загрузка файла через Telegram в локальном окружении;
- открытие списка файлов в вебе;
- скачивание владельцем;
- отказ скачивания чужого файла;
- адаптивность списка и медиатеки на мобильной ширине.

## 13. Зоны ответственности агентов

- `architect-reviewer` — проверяет границы компонентов, направление зависимостей, data ownership, сложность и риски расширения.
- `django-developer` — владеет Django models, views, admin, URL, ORM-запросами и миграциями.
- `backend-developer` — владеет сервисным слоем, контрактами поведения, ошибками, транзакциями и правами доступа.
- `aiogram-engineer` — владеет bot runtime, routers, middleware, обработкой Telegram-файлов и идемпотентностью update.
- `sql-pro` — проверяет модель данных, индексы, фильтры, сортировки, пагинацию и миграционные риски.
- `python-pro` — проверяет упаковку, entry points, импорты, типовые контракты и тестируемость.
- `docker-expert` — владеет Dockerfile, Docker Compose, правами runtime-пользователя, healthcheck и секретами окружения.
- `platform-engineer` — проверяет локальный developer workflow, конфигурацию окружений, запуск сервисов и operational path.
- `api-documenter` — описывает API только после появления реальных endpoints и схем.
- `docs-researcher` — сверяет спорные детали Django, aiogram, Docker или API по официальной документации.
- `code-reviewer` — проводит итоговую проверку изменений перед закреплением этапа.

## 14. Первый исполнимый срез разработки

Первый срез создаёт минимальную Django-основу без Telegram-интеграции:

1. Создать Django project `homehub`.
2. Создать приложения `accounts` и `files`.
3. Добавить модель `FileAsset`.
4. Добавить admin для `FileAsset`.
5. Реализовать `FileAccessService`.
6. Реализовать список файлов для авторизованного пользователя.
7. Реализовать защищённый download endpoint.
8. Добавить базовые тесты прав доступа.

После этого подключается второй срез:

1. Добавить aiogram bot runtime.
2. Реализовать `/help`, `/status`, `/files`.
3. Реализовать загрузку файла через Telegram в `FileIngestionService`.
4. Добавить проверку Telegram identity.
5. Проверить локальный long polling.

## 15. Открытые решения перед кодированием

- Использовать ли только Django templates на первом экране или сразу добавить небольшой JSON API для будущего UI.
- Делать ли логическое удаление обязательным в MVP или начать с физического удаления после проверки прав.
- Генерировать ли thumbnails в первом срезе или показывать только исходные изображения/видео без отдельной обработки.
- Нужен ли отдельный `apps/media/` сразу или оставить медиатеку как view поверх `FileAsset`.
